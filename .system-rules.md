# System Rules Registry

This registry tracks behavioral rules, scraper parameters, and optimization constraints compiled and abstracted over execution runs.

- **R-101**: Never run geocoding APIs without local cache layers to prevent redundant coordinate billing.
- **R-102**: Enrichments using Gemini 2.5 Flash must use strict JSON schemas (structured output) to prevent formatting/parsing failures on special characters.
- **R-103**: When deploying new directory subdomains, they must follow the format `niche-city.com` or `niche-state.com` and must be cataloged in `directory-registry.md` before going live.
- **R-104**: Express 5 wildcards must be bound inside matching parameter brackets (e.g. `{*wildcard}`) to ensure compatability with path-to-regexp v8.
- **R-105**: Production container configurations must bind Express API servers to port 8080 (`PORT=8080`) to satisfy GCP health check probes.
- **R-106**: Any scraping activity must be small, self-contained, iterative, and non-duplicate. Do not run large all-at-once pulls. Test and iterate on smaller runs first to verify API mechanisms and dataset structures, securing a definable result before scaling up.
- **R-107**: All AnswerThePublic keyword searches must be executed via the shared CLI script (`02-workbench/answerthepublic/scripts/fetch-atp.py`) with search terms passed as arguments. Results must be cached in `02-workbench/answerthepublic/cache/` to prevent redundant credit billing, enforcing a limit of 10 or fewer searches per directory launch.
- **R-108**: If a PDF file is available online, always download it and parse it locally using a Python script (e.g. via `fitz`/PyMuPDF or `pypdf`) instead of using browser screenshots or browser-based text extraction. This minimizes token consumption and prevents browser navigation loops.
- **R-109**: Fictional, mock, synthetic, or realistic-looking placeholder data is strictly prohibited for regulatory fields (such as permit numbers or license statuses). All compliance and listing metadata must be derived from verified public registries and actual records.
- **R-110**: Unclaimed or basic listings on all directory sites must never display negative badges or labels (e.g. "Unclaimed" or "Unverified") to the public user, as it incorrectly implies the business itself is unverified. Design directories to move from neutral (regular listings with no flags/badges) to positive (featured/claimed/premium listings only).
- **R-111**: Programmatic geocoding and scraper actors running on Apify must retrieve seeds and configurations via Key-Value store IDs to maintain consistent data flows, and require local/cloud API key wallet pre-funding (e.g., OpenAI, Google Places API).
- **R-112**: When troubleshooting Apify actor compilation or runtime errors, copy raw logs to the LLM agent environment for rapid code diagnostics.
- **R-113**: Strictly maintain the workspace directory structure for scripts and scratch work. Generalizable scripts go in `/scripts`. Project-specific scripts go in the corresponding workbench folder (`02-workbench/<project>/scripts`). Temporary scratch work must be stored in `/scratch`. The root directory must remain clean of any scripts or scratch files.
- **R-114**: Git Commit Threshold: Always ask, "would this take more than 10 to 15 minutes to recreate if we lost our current version?" If yes, then commit the updates to version control immediately.
- **R-115**: Avatar Adjacency Clusters: When investigating a new niche directory idea, always map out the "adjacency cluster" for that specific buyer persona (avatar). Niches that allow us to build multiple interconnected directories solving different problems for the exact same avatar (e.g. Restaurant Owner -> Grease Traps + Hood Cleaning + Fire Inspections) receive a strong positive viability modifier.
- **R-116**: Scale vs. Frequency Equation: When evaluating the "Recurring Need" of a niche, low-frequency events (e.g., a 10-year replacement cycle) must be offset by massive geographic scale (nationwide targeting) to maintain a sustainable lead volume. Conversely, high-frequency compliance events (e.g., 90-day pumping cycles) can survive on highly localized, single-state targeting. Equation: `Sustainable Volume = Purchase Frequency × Geographic Scale`.
- **R-117**: The 7-Layer Hierarchy Framework: All programmatic directory databases must be mapped into seven rigid layers before any code is written: (1) Core Location, (2) Import Requirements, (3) Domain-Specific Fields, (4) AI Enrichment, (5) Google Places Data, (6) Data Freshness/Tracking, and (7) Monetization.
- **R-118**: AEO Content Mandate: To optimize for AI search engines (Voice/LLM crawlers), all enriched directory profiles must generate 20 granular Q&A blocks. The first sentence of every answer must directly and completely answer the question as if spoken out loud, avoiding summary statements or bulleted lists.
- **R-119**: Anti-Thin-Content Gate: Programmatic listings must pass a mandatory quality gate before hitting the production environment. A listing must have a minimum of 100 words of unique content, a populated primary domain-specific field (e.g., license number), and active imagery to prevent indexation penalties.
- **R-120**: Frontend Speed & Static Generation: High-urgency local intent queries demand sub-100ms Time to First Byte (TTFB). Monolithic backends are disqualified. Frontends must use Static Site Generation (SSG) via Astro or Incremental Static Regeneration (ISR) via Next.js to pre-build top-tier pages while deferring long-tail generation.
- **R-121**: Polymorphic JSONB Data Architecture: To support "Avatar Adjacency" expansion (e.g., adding propane or septic vendors to the same database structure), production databases (like Supabase PostgreSQL) must store core universal fields natively (Name, City, State) and store industry-specific details in a flexible `JSONB` bucket to avoid breaking schema migrations.
- **R-122**: Decoupled Staging Pipelines: Computationally heavy LLM enrichment loops (e.g., using Gemini 2.5 Flash) must never execute directly on live production tables. Extraction data (local SQLite dumps) must be routed into a cloud `staging` schema where cleaning and AI enrichment occur, syncing to production only after the Anti-Thin-Content Gate is passed.
- **R-123**: Strict Key/Slug Standardization: To ensure predictable technical infrastructure, the human-readable niche name must always be converted into a clean, lower-case slug (e.g., `water-well-drillers`). This slug must universally anchor all corresponding KV store names and file artifacts using standardized suffixes (`[slug]_seed.csv`, `[slug]_enriched.csv`, `[slug]_pending.csv`) to prevent data mapping failures.
- **R-124**: Two-Tier LLM Enrichment & Phase 1 Validation: To maximize token efficiency during pipeline development, initial AI enrichments must be limited to a "Phase 1" run of exactly 7 questions. These must be divided into a Tier 1 Universal Baseline (5 core transactional intents like hours, cost, access) and a Tier 2 High-Urgency Filter (2 niche-specific questions directly addressing the primary emotional pain point or conversion barrier). Only after validating this 7-question batch should pipelines scale to the full 20-question schema.
- **R-125**: Absolute Database Key Consistency: Directory databases and import functions require strict structural alignment. Before building an extraction pipeline, the total field array (Core, Import, Domain, AI, Places, Freshness, Monetization) must be calculated (e.g., 98 fields). Extraction code, AI prompts, and database schemas must use identically matched object keys to prevent silent data-dropping during syncs.
- **R-126**: The Agent-Accelerated 4-Phase Factory: All new directory projects must proceed strictly through 4 phases: P1 Selection, P2 Agentic Discovery (Prototyping), P3 Engineering (Data Pipeline), and P4 Antigravity Deployment. Workbench files must use prefixes (`p1_`, `p2_`, `p3_`, `p4_`) to indicate their phase rather than creating deeply nested directory trees.
